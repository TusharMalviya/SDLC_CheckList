Developer Checklist – CRONOS

Standard development process to be followed for every Jira story / task

Version 1.0  |  Mar 2026

Jira / Story Details

Overall Progress
0 %
1 Jira Requirement Understanding
  • Read the Jira story / task in full detail
  • Understand acceptance criteria and definition of done
  • Clarify open questions with the Business Analyst / Product Owner
  • Link related Jira tickets (dependencies, sub-tasks, bugs)
  • Estimate story points / effort and update Jira
2 FSD (Functional Specification Document) Understanding
  • Locate and read the relevant FSD sections
  • Cross-check the FSD against the Jira story scope
  • Identify business rules, validations, and data flows described
  • Note any ambiguities; raise them with the BA / Tech Lead
  • Confirm FSD version used is the latest approved version
Deliverable: annotated FSD reference saved in release track folder
3 Initial Analysis of the Requirement
  • Identify modules / services impacted by the change
  • Review existing code in the impacted areas
  • List upstream and downstream dependencies
  • Identify DB schema changes required (if any)
  • Highlight risks and assumptions
  • Create Initial Impact Analysis file and save to release track folder
Deliverable: Initial_Impact_Analysis_<JiraID>.xlsx / .docx
4 Work Breakdown Structure (WBS)
  • Break the requirement into tasks (analysis, design, code, test, review)
  • Assign effort estimate (hours/days) to each task
  • Plan tasks on a daily basis with start and end dates
  • Identify blockers / dependencies between tasks
  • Update WBS daily with actual vs planned progress
  • Share WBS with Tech Lead for review and approval
Deliverable: WBS_<JiraID>.xlsx – updated daily
5 Technical Design Document (TDD)
  • Start TDD as soon as development begins (living document)
  • Document high-level architecture / design decisions
  • Include class / sequence / flow diagrams where needed
  • Document DB changes (DDL scripts, ER diagram)
  • Capture API contracts / service interfaces changed
  • Keep TDD updated throughout development
Deliverable: TDD_<JiraID>_v1.0.docx

Note: TDD is a living document – continue updating through Step 10.

6 Code Implementation
  • Create a feature branch from the correct base branch
  • Follow project coding standards and naming conventions
  • Implement the changes as per TDD and FSD
  • Add meaningful comments / Javadoc where applicable
  • Ensure no hardcoded values (use configuration / constants)
  • Handle exceptions and edge cases properly
  • Commit incrementally with meaningful commit messages
  • Resolve merge conflicts with the base branch
  • Verify the build is clean (no compile errors / warnings)
7 Unit Testing (UT)
  • Write unit tests for all newly developed code paths
  • Write / update tests for all impacted existing areas
  • Cover positive, negative, and boundary test cases
  • Achieve minimum required code coverage (check project threshold)
  • Run all existing tests – ensure none are broken
  • Capture test execution screenshots / reports as proof
  • Create UT Document with all test cases and proof screenshots
Deliverable: UT_Document_<JiraID>.xlsx / .docx (includes test cases + screenshots)
8 Code Review
  • Self-review the code before scheduling the review session
  • Schedule a code review call with the reviewer(s)
  • Record the review session (Teams / Zoom recording)
  • Walk the reviewer through the changes and the TDD
  • Document all review comments and action items
  • Address all comments and get reviewer confirmation
  • Save recording link / file to release track folder
Reviewers: Prasanna Shilpa Vishal
Review session must be recorded
9 Full Detailed Impact Analysis (Post-Development)
  • List all files / classes / tables changed with the reason
  • Document all new components / services introduced
  • Capture all impacted downstream / upstream systems
  • Describe regression risk areas and mitigation
  • Include DB migration scripts and rollback plan
  • Compare against Initial Impact Analysis (Step 3) – note deviations
Deliverable: Full_Impact_Analysis_<JiraID>_PostDev.xlsx / .docx
10 TDD & Impact Analysis Review
  • Finalise the TDD with all implementation details
  • Present TDD & Full Impact Analysis to Tech Lead / reviewer
  • Incorporate review feedback into TDD and Impact Analysis
  • Get TDD & Impact Analysis approved (at least one approver signature)
Reviewers: Prasanna Shilpa Vishal
Deliverable: TDD_<JiraID>_Final.docx | Full_Impact_Analysis_Final.docx
11 Sign-Off Mail
  • Request sign-off mail from the reviewer after code review is complete
  • Mail must include Jira ID, story title, review date, and reviewer name
  • Mail must confirm all review comments have been resolved
  • Save the sign-off mail (as .msg / .eml / PDF) to release track folder
Sign-off from: Prasanna Shilpa Vishal
Sign-off mail is mandatory before release
12 Place All Documents in Release Track Folder
  • Initial Impact Analysis file
  • WBS Excel file (final version)
  • TDD – Final version
  • UT Document with all test proofs
  • Code review session recording (link or file)
  • Full Post-Development Impact Analysis
  • Sign-off mail from reviewer
  • FSD reference (annotated, if applicable)
  • Confirm folder path follows the release naming convention
All documents verified and saved – Release ready
